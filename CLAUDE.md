## Development Philosophy

### Test-Driven Development (TDD)

- 原則としてテスト駆動開発（TDD）で進める
- 期待される入出力に基づき、まずテストを作成する
- 実装コードは書かず、テストのみを用意する
- テストを実行し、失敗を確認する
- テストが正しいことを確認する
- その後、テストをパスさせる実装を進める
- 実装中はテストを変更せず、コードを修正し続ける
- すべてのテストが通過するまで繰り返す

## Subagent Model Tiering

サブエージェント（Agent tool）へタスクを委譲するときは、タスクの性質に応じてモデルを使い分ける。
メインセッションはオーケストレーターに徹し（計画・分解・統合・最終判断）、実装などの手を動かす作業はサブエージェントに委譲する。

参考: https://zenn.dev/yui/articles/740da24e9ee419

### モデル使い分け基準

| タスクの種類 | モデル | 例 |
|---|---|---|
| 計画・分解・統合・最終判断 | メインセッションで実施（委譲しない） | タスク分解、結果の統合、トレードオフ判断 |
| スコープ検討・技術設計 | `opus` | 実装前の方針決定、アーキテクチャ設計 |
| 検証・コードレビュー | `opus` | エッジケース発見、品質チェック |
| 実装・テスト作成・デバッグ | `sonnet` | コード書き、テスト実装、バグ修正 |
| 単純・機械的な作業 | `haiku` | ファイル整理、一括検索、ログ抽出、定型的な変換 |

### 運用ルール

- Agent tool 呼び出し時に `model` パラメータで明示的に指定する
- 「考える人／手を動かす人」の二分ではなく、役割を細かく分けて委譲の精度を上げる
- 迷ったら一段安いモデルを試し、品質不足なら上位モデルでやり直す
- あいまいでトレードオフの多い判断はサブエージェントに投げず、メインセッションで行う
